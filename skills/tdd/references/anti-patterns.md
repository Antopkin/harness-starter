# TDD Anti-Patterns

## 1. Horizontal Slicing (главный антипаттерн)

**НЕ ДЕЛАЙ:** написать все тесты сначала, потом писать всю реализацию.

```
WRONG (horizontal):
  RED:   test_1, test_2, test_3, test_4, test_5
  GREEN: impl_1, impl_2, impl_3, impl_4, impl_5
```

Что получится:
- Тесты, написанные пачкой, проверяют **воображаемое** поведение, не реальное.
- Кончится тестированием формы (структуры данных, сигнатур функций), а не пользовательского поведения.
- Тесты становятся нечувствительными к реальным изменениям: проходят, когда поведение сломалось; падают, когда поведение в порядке.
- Ты обгоняешь свои фары: фиксируешь структуру тестов до понимания реализации.

**ПРАВИЛЬНО:** vertical slicing через tracer bullets. Один тест → одна реализация → повтор.

```
RIGHT (vertical):
  RED→GREEN: test_1 → impl_1
  RED→GREEN: test_2 → impl_2
  RED→GREEN: test_3 → impl_3
```

Каждый следующий тест отвечает на то, что ты узнал из предыдущего цикла. Поскольку код только что написан, ты точно знаешь, какое поведение важно и как его проверить.

См. `vertical-slicing.md` для подробностей.

## 2. Тестирование имплементации, не поведения

**НЕ ДЕЛАЙ:** мокать внутренних коллабораторов, тестировать private методы, проверять через побочные каналы (читать из БД напрямую вместо использования интерфейса).

Sign of trouble: тест ломается при рефакторинге, хотя поведение не изменилось. Если переименование внутренней функции ломает тест — этот тест проверял реализацию, не поведение.

```python
# BAD: тестирует имплементацию (мокает внутреннего коллаборатора)
def test_checkout_calls_payment_service():
    cart = Cart()
    mock_payment = Mock(spec=PaymentService)
    checkout(cart, mock_payment)
    mock_payment.process.assert_called_once_with(cart.total)

# GOOD: тестирует поведение через public interface
def test_checkout_returns_confirmed_status():
    cart = Cart()
    cart.add(product)
    result = checkout(cart, payment_method)
    assert result.status == "confirmed"
```

См. `behavior-vs-implementation.md`.

## 3. Speculative tests (на будущее)

**НЕ ДЕЛАЙ:** добавлять тесты для функциональности, которая ещё не нужна. "Вдруг пригодится."

Каждый тест — обязательство: его надо поддерживать, рефакторить вместе с кодом, объяснять команде. Speculative tests превращаются в технический долг. Пиши только тот тест, который описывает поведение, нужное прямо сейчас. Тег спеки (`# spec: <id>`, см. `spec-phase.md`) ловит это: тест без трассируемого AC/EC/ERR — кандидат в speculative.

---

## 4. Refactor под красным

**НЕ ДЕЛАЙ:** рефакторить, когда тесты не зелёные. Сначала pass, потом refactor.

Если попытаешься рефакторить под красным — не будешь знать, что сломалось: новый refactor или старый bug. Получи зелёное состояние любыми средствами (хоть hardcoded return), потом улучшай.

---

## 5. Многоассертный тест

**НЕ ДЕЛАЙ:** проверять несколько независимых поведений в одном тесте. Один тест — одно логическое утверждение.

```python
# BAD: проверяет несколько вещей
def test_checkout():
    assert cart.total == 100
    assert checkout(cart).status == "confirmed"
    assert send_email.called

# GOOD: разбито на 3 теста
def test_cart_total_sums_items(): ...
def test_checkout_confirms_valid_cart(): ...
def test_checkout_sends_confirmation_email(): ...
```

Когда падает один из 3 раздельных — точно знаешь что сломалось. Когда падает многоассертный — копаешь.
