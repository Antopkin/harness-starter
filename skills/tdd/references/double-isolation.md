# Double Isolation + Contradiction Detector

Чтобы не было «подсознательного жульничества» — когда автор теста уже знает реализацию, а автор кода подгоняет под подсмотренный intent спеки — каждая половина цикла идёт в **своём изолированном контексте субагента**. Это симметричное расширение прежней односторонней изоляции (раньше изолировался только GREEN).

Главный принцип: **каждый контекст знает ровно одну сторону контракта.** Тест-райтер видит спеку, но не реализацию. Разработчик видит тест, но не спеку.

---

## Два контекста: что выдаём, что скрываем

### Test-writer context (пишет падающий тест)

| Получает | Лишён доступа |
|----------|---------------|
| SPEC (нумерованные AC/EC/ERR) | путь к файлу реализации |
| путь к файлу тестов (на запись) | возможности читать `src/`, impl-модули |
| соглашение о тегах (`# spec: AC-1`) | любого знания о текущей реализации |
| команда запуска pytest | — |

Инструкция: «Напиши **один** падающий тест из спеки. Каждый тест ссылается на свой критерий (`# spec: AC-N`). Тест проверяет поведение через публичный интерфейс. Убедись, что тест FAIL до возврата. Не смотри и не предполагай реализацию.»

Если импортируемого модуля ещё нет — создай пустую заглушку (пустой `def`/класс), чтобы тест падал на ассерте, а не на `ImportError`. Отметь это в выводе.

### Developer / GREEN context (пишет минимальный код)

| Получает | Лишён доступа |
|----------|---------------|
| ТОЛЬКО текущий тест (**read-only!**), с **вырезанным тегом** `# spec:` | путь к спеке |
| команду прогнать остальные тесты (видит pass/fail, не их тела) | прозаический текст требований |
| путь/имя файла реализации (на запись) | накопленный файл тестов целиком |
| вывод pytest (сообщение об ошибке) | — |

Инструкция: «Напиши **минимальный** код, чтобы прошёл ЭТОТ тест. НЕ трогай тестовые файлы. Hardcoded-значения допустимы. Не выдумывай поведение сверх теста. Если, делая этот тест зелёным, ты роняешь ранее зелёный тест — НЕ подгоняй: верни `cross_ac_break` (см. ниже).»

**Почему разработчику дают только текущий тест, без тега и без накопленного файла.** Изоляция «один тест глубиной» протекает, если на вход идёт весь растущий файл тестов: тег `# spec: AC-1` — это указатель на критерий, а набор тегов + ассерты по нескольким тестам позволяют reverse-engineer структуру спеки и кодить под intent, а не под тест. Поэтому main **вырезает тег** из копии для разработчика (карту тег↔тест держит у себя) и отдаёт **только текущий тест**, а остальные — лишь командой запуска (pass/fail), не телами. Разработчик видит «вот вход, вот ожидаемый выход» — и удовлетворяет ровно это.

```python
# ХОРОШО — developer видит поведение и ассерт, не прозу требования:
def test_format_file_size_kilobytes():
    # spec: AC-1
    assert format_file_size(1024) == "1 KB"

# ПЛОХО — проза требования вшита в тест, изоляция протекает:
def test_format_file_size_kilobytes():
    """AC-1: согласно ТЗ от заказчика, для совместимости с legacy-биллингом
    размеры всегда в бинарных KB/MB, заказчик отдельно просил ..."""
    assert format_file_size(1024) == "1 KB"
```

### Оркестрация

- **Main context** держит спеку и координирует: SPEC, PLAN, RED-launch, REFACTOR, COMMIT, SUMMARY.
- **Test-writer субагент** — RED (пишет тест из спеки, blind к impl).
- **Developer субагент** — GREEN (пишет код из теста, blind к спеке).

Без субагентов: работай в main, но строго соблюдай две границы — на RED не подсматривай реализацию, на GREEN не сверяйся со спекой, только с тестом.

## `spec_defect` — сигнал противоречия

Если спека внутренне противоречива, нереализуема или недоопределена — НЕ компенсируй догадками и не ослабляй тесты. **Останови и подними `spec_defect`.**

### Триггеры

1. **Contradictory criteria** — два AC/EC/ERR требуют взаимоисключающего поведения (AC-1 «возвращает синхронно» vs AC-3 «возвращает корутину»).
2. **Missing precondition** — тесты предполагают форму ввода/зависимость, которую спека не определяет, и разумного дефолта нет.
3. **Unimplementable constraint** — критерий нереализуем как заявлено (AC «уложиться в <10ms», но требует сетевого round-trip без кэша).
4. **Under-specification** — один критерий удовлетворяется несколькими существенно разными реализациями, и выбрать нельзя без угадывания intent.

### Что делать при триггере

1. STOP. Не пиши и не меняй ни один файл реализации.
2. Не меняй тестовый файл (правило неизменности тестов держится и здесь).
3. Верни сигнал:

```
spec_defect: true
spec_defect_reason: "<1-2 предложения, назвать конкретные id>"
status: spec_defect
files_modified: []
summary: "no implementation attempted — spec_defect"
```

### Что НЕ дефект спеки (обрабатывай штатно)

- Тест падает из-за бага в реализации → чини реализацию.
- Критерий сложный, но реализуемый → продолжай.
- Незнакомая библиотека → читай существующий код проекта.
- Не подумал про edge case → подумай лучше, не эскалируй.

### Анти-паттерны (никогда)

- Ослаблять ассерты, чтобы протолкнуть дефектный тест.
- Lookup-таблица / hardcode только под входы теста, чтобы обойти противоречие спеки.
- Выдумывать поведение, которого спека не определяет, и тихо надеяться, что угадал.

## Worked example: premium-дашборд (должен дать `spec_defect`, не зелёные тесты)

Спека содержит два AC:

```
AC-2: неоплативший юзер НЕ имеет доступа к premium
AC-5: после регистрации юзер сразу видит premium-дашборд
```

Только что зарегистрированный неоплативший юзер по AC-5 обязан видеть premium, а по AC-2 обязан его НЕ видеть. Для одного и того же состояния (just-registered, unpaid) требования взаимоисключающи.

**Правильный выход — Coherence check (см. `spec-phase.md`) ловит это до RED:**

```
spec_defect: true
spec_defect_reason: "AC-2 vs AC-5: только что зарегистрированный неоплативший
  юзер одновременно обязан НЕ видеть premium (AC-2) и сразу видеть
  premium-дашборд (AC-5) — взаимоисключающе для одного состояния."
status: spec_defect
```

Halt, эскалация человеку на разрешение. **Не** писать `if just_registered: show_premium`, не делать lookup-таблицу, не «выбирать» одно из AC молча.

## 2-strike rule (латентные противоречия в GREEN)

Coherence check ловит явные противоречия статически. Но бывают **латентные** — видны только при попытке сделать зелёным.

Симптом: тест нельзя сделать проходящим, не сломав другой уже-зелёный тест, привязанный к **другому** AC. Это значит, что два AC незаметно конфликтуют через реализацию.

**Кто считает strikes — критично.** GREEN идёт в свежем blind-субагенте каждый цикл; такой контекст не помнит прошлый strike, и «New PASS, old FAIL» каждый раз выглядит как обычное «почини реализацию» — счётчик, живущий внутри GREEN, не сработает никогда, и противоречие будет вечно «чиниться» test-by-test, выдавая зелёное на сломанной спеке. Поэтому strike — это **возвращаемый токен, который считает main**, а не нарратив внутри GREEN:

1. GREEN-субагент, обнаружив, что ради зелёного для `AC-i` падает ранее зелёный тест `AC-j`, НЕ подгоняет тест и **возвращает** структурный сигнал:
   ```
   cross_ac_break: { broke: AC-j, while_greening: AC-i }
   ```
   и отдаёт управление, не форсируя зелёное.
2. **Main** держит счётчик strikes по парам AC через все циклы (blind-контекст GREEN его держать не может). Любой `cross_ac_break` — это strike, независимо от счётчика «3 GREEN-попытки». На **2-м** strike по одной паре main делает **STOP, emit `spec_defect`**:

```
spec_defect: true
spec_defect_reason: "AC-i vs AC-j: green для одного стабильно ломает
  green для другого — латентное противоречие после 2 strikes."
status: spec_defect
```

Счётчик живёт в main в пределах одного `/tdd` цикла, стартует с 0, не сбрасывается между циклами. После 2-го strike — эскалация человеку, без третьего автозахода: спека фундаментально неоднозначна.
